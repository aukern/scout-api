## 1. Manage collections

**Issue:** [#9](https://github.com/aukern/scout-api/issues/9)

### What to build
The ability to create, list, and delete **Collections** — named partitions of knowledge.
Every Source, Search, Question, and Session is scoped to exactly one Collection; this slice
is the foundation that isolation (ADR 0001) rests on. A Collection name is unique. Deleting a
Collection removes the knowledge it contains (its Sources and their Chunks).

### Acceptance criteria
- [ ] A Collection can be created with a unique name; duplicate names are rejected
- [ ] Collections can be listed
- [ ] A Collection can be deleted, cascading to its Sources and Chunks
- [ ] Collection names follow the glossary (avoid "namespace/tenant/index/corpus" in the API)

### Blocked by
None — can start immediately.

- Operation: CREATE
- Specializations: `[py:postgres]` `[py:api]`

## 2. Ingest a source into a collection

**Issue:** [#10](https://github.com/aukern/scout-api/issues/10)

### What to build
Accept a **Source** — a URL or an uploaded file — into a Collection. The call creates the
Source in **pending**, stores the uploaded blob in S3-compatible object storage (a URL Source
records its address; fetching is deferred to processing), enqueues a background processing job,
and returns the Source id and status immediately (write-only; no waiting for processing).

A Source is **unique within its Collection by origin**: re-ingesting the same URL/file refreshes
the existing Source in place (drops its old Chunks, re-queues processing) rather than creating a
duplicate (CONTEXT.md; ADR 0001 consequences).

### Acceptance criteria
- [ ] POST a URL or file to a Collection creates a Source with status `pending` and returns its id
- [ ] An uploaded file's bytes are stored in object storage; a URL Source records its origin
- [ ] A processing job is enqueued (the response does not wait for it)
- [ ] Re-ingesting the same origin in the same Collection refreshes the existing Source in place — no duplicate Source, old Chunks removed
- [ ] Ingesting the same origin into a *different* Collection creates a separate Source (uniqueness is per-Collection)

### Blocked by
- Slice 1 (Manage collections)

- Operation: CREATE
- Specializations: `[py:postgres]` `[py:api]` `[py:s3]` `[py:arq]` `[py:redis]` `[py:mcp]`
- Slice pattern: `[pattern:ingest]`

## 3. Process a source into searchable chunks

**Issue:** [#12](https://github.com/aukern/scout-api/issues/12)

### What to build
The background worker that turns a pending Source into searchable knowledge: move it to
**processing**, split its content into **Chunks**, generate an **Embedding** per Chunk via the
configured model, store the vectors, then mark the Source **ready**. If processing fails, mark it
**failed** (and leave it excluded from Search). This is an internal worker — no HTTP endpoint.

**Decision (from prototype):** the embedding dimension is **derived from the model at setup**, not
hardcoded — the `chunks.embedding vector(N)` dimension must follow the configured embedding model
(e.g. 768 for `nomic-embed-text`, 1536 for `text-embedding-3-small`).

### Open questions to resolve (from FINDINGS)
- **Chunking strategy** — the prototype used a naive ~200-char packer; production wants a
  token-aware splitter with overlap. Pick and document the strategy.
- **Embedding dimension as config** — schema dimension must track the model; decide how the
  dimension is set/migrated when the model changes.
- **Failed-source retry policy** — do failed Sources retry automatically, on demand, or never?

### Acceptance criteria
- [ ] A queued Source transitions `pending → processing → ready` on success
- [ ] Content is split into Chunks; each Chunk gets an Embedding from the configured model
- [ ] The vector column dimension matches the embedding model (not hardcoded)
- [ ] On failure the Source becomes `failed` and produces no searchable Chunks
- [ ] Re-processing a refreshed Source replaces its Chunks (no stale/duplicate vectors)

### Blocked by
- Slice 2 (Ingest a source)

### Pipeline tags
- Operation: CREATE
- Specializations: `[py:postgres]` `[py:vector]` `[py:litellm]` `[py:arq]` `[py:redis]`
- Slice pattern: `[pattern:command]`

## 4. Browse sources & their status

**Issue:** [#13](https://github.com/aukern/scout-api/issues/13)

### What to build
Read-only visibility into the Sources of a Collection and each Source's lifecycle status
(`pending → processing → ready → failed`). This is what lets an agent confirm a Source is
**ready** before relying on Search (the CONTEXT.md example dialogue: "search returned nothing
because the Source wasn't ready yet").

### Acceptance criteria
- [ ] List the Sources in a Collection, each with its current status
- [ ] Fetch a single Source's status and metadata (origin, status)
- [ ] Listing is scoped to one Collection (a Collection never shows another's Sources)

## 5. Semantic search within a collection

**Issue:** [#15](https://github.com/aukern/scout-api/issues/15)

### What to build
A **Search**: free-text in, ranked **Results** out, scoped to one Collection. Embed the query,
run a pgvector cosine nearest-neighbour over the Collection's Chunks, and return Results ordered
by relevance — each carrying its score and originating Source. Only Chunks from **ready** Sources
are searchable. Identical repeated searches are served from cache.

**Guarantees (asserted in the prototype):** results never leak across Collections (isolation,
ADR 0001); a Source that isn't `ready` contributes nothing.

### Acceptance criteria
- [ ] A query returns Chunks ranked by semantic relevance (not keyword match), each with score + Source
- [ ] Results are restricted to the queried Collection — no cross-Collection leakage
- [ ] Only Chunks from `ready` Sources appear
- [ ] Repeated identical searches are served from cache; cache invalidates when the Collection's content changes
- [ ] A Search never synthesizes — it only retrieves

### Blocked by
- Slice 3 (Process a source into searchable chunks)

### Pipeline tags
- Operation: CREATE
- Specializations: `[py:postgres]` `[py:vector]` `[py:litellm]` `[py:api]` `[py:redis]` `[py:mcp]`
- Slice pattern: `[pattern:query]`

> Note: deliberately **no `[py:arq]`** despite the api+LLM auto-rule — a Search must return
> synchronously, and the single query-embedding call is fast (per the registry's synchronous-read
> exception).

## 6. Ask a question, stream a cited answer

**Issue:** [#16](https://github.com/aukern/scout-api/issues/16)

### What to build
A **Question**: scoped to one Collection, it retrieves the most relevant Chunks, synthesizes an
**Answer** with an LLM, and **streams** the answer token-by-token with **Citations** back to the
Source(s) it drew from. If retrieval finds nothing useful, the Answer says the context is
insufficient rather than inventing one.

**Prototype reference** — the synthesis prompt that produced correctly grounded, cited answers:
> "Answer the question using ONLY the numbered context. Cite sources inline as [1], [2]. If the
> context is insufficient, say so."

### Open question to resolve (from FINDINGS)
- **Citation granularity** — should a Citation point to the Source, or to the specific Chunk
  within the Source? CONTEXT.md says "the specific Source (and Chunk)"; pick the precise contract.

### Acceptance criteria
- [ ] A Question scoped to a Collection retrieves relevant Chunks and synthesizes an Answer
- [ ] The Answer streams to the caller incrementally (token/event by event)
- [ ] The Answer carries Citations to the Sources it used
- [ ] The Answer is grounded only in retrieved context; insufficient context is stated, not fabricated
- [ ] Retrieval is Collection-scoped (same isolation guarantee as Search)

### Blocked by
- Slice 5 (Semantic search) — reuses Collection-scoped retrieval

### Pipeline tags
- Operation: CREATE
- Specializations: `[py:postgres]` `[py:vector]` `[py:litellm]` `[py:websocket]` `[py:mcp]`
- Slice pattern: `[pattern:query]`

> Note: streaming is handled by `[py:websocket]` (the slow LLM work *is* the stream), so no
> `[py:arq]` is needed here. `[py:websocket]` keeps the underlying `query` pattern.

## 7. Research sessions

**Issue:** [#11](https://github.com/aukern/scout-api/issues/11)

### What to build
A **Session** is a research workspace opened against exactly one Collection that records an
activity trail: every Search run and every Question asked (with its Answer). Recording is
**opt-in** — a Search or Question is recorded only when it names a Session; otherwise it runs
transiently and leaves no trace. Supports open / list / get / close.

Slices 5 (Search) and 6 (Q&A) record into a Session when a session id is supplied — this slice
owns the Session lifecycle and the recording mechanism they call.

### Open question to resolve (from FINDINGS)
- **Session retention** — how long are Sessions and their activity kept; is there a cleanup policy?

### Acceptance criteria
- [ ] A Session can be opened against exactly one Collection
- [ ] A Search/Question that names a Session is recorded in its activity trail; one that doesn't is not
- [ ] All activity in a Session is scoped to that Session's Collection
- [ ] Sessions can be listed, fetched, and closed

### Blocked by
- Slice 1 (Manage collections)

### Pipeline tags
- Operation: CREATE
- Specializations: `[py:postgres]` `[py:api]`
- Slice pattern: `[pattern:crud]`

## 8. Save answers as briefs

**Issue:** [#14](https://github.com/aukern/scout-api/issues/14)

### What to build
Within a Session, save an **Answer** (its text plus its Citations) as a **Brief** — the durable,
kept version of an otherwise transient Answer. A Session has many Briefs; a Brief belongs to
exactly one Session. Supports saving and listing Briefs in a Session.

### Acceptance criteria
- [ ] An Answer (text + Citations) can be saved as a Brief within a Session
- [ ] Briefs in a Session can be listed
- [ ] A Brief belongs to exactly one Session; it cannot exist outside one
