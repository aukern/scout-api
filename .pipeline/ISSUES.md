## 2. Ingest a source into a collection

Status: open
**Issue:** [#18](https://github.com/aukern/scout-api/issues/18)

### What to build
Accept a **Source** (URL or uploaded file) into a Collection: create it `pending`, store an
uploaded blob in S3 (a URL records its address; fetch deferred to processing), enqueue a
processing job, return id+status immediately (write-only). Re-ingesting the same origin in the
same Collection refreshes in place (drops old Chunks) — no duplicate (ADR 0001).

### Acceptance criteria
- [ ] POST a URL or file → Source `pending`, returns id
- [ ] Uploaded file bytes stored in object storage; URL Source records its origin
- [ ] Processing job enqueued (response does not wait)
- [ ] Re-ingesting same origin refreshes in place — no duplicate, old Chunks removed
- [ ] Same origin in a different Collection is a separate Source (uniqueness per-Collection)

## 3. Process a source into searchable chunks

Status: open
**Issue:** [#20](https://github.com/aukern/scout-api/issues/20)

### What to build
Background worker turning a pending Source into searchable knowledge: `processing` → chunk →
embed each Chunk → store vectors → `ready`; on failure → `failed` (excluded from Search).
Internal worker, no HTTP endpoint. **Decision (prototype):** embedding dimension is derived from
the model, not hardcoded — `chunks.embedding vector(N)` follows the configured model.

### Open questions (FINDINGS)
- Chunking strategy (prototype used a naive ~200-char packer; production wants token-aware + overlap)
- Embedding dimension as config / migration when the model changes
- Failed-source retry policy

### Acceptance criteria
- [ ] `pending → processing → ready` on success
- [ ] Content split into Chunks; each gets an Embedding from the configured model
- [ ] Vector column dimension matches the model (not hardcoded)
- [ ] On failure → `failed`, no searchable Chunks
- [ ] Re-processing a refreshed Source replaces its Chunks

### Blocked by
- slice 2 (Ingest a source into a collection)

### Pipeline tags
- Operation: CREATE
- Specializations: `[py:postgres]` `[py:vector]` `[py:litellm]` `[py:arq]` `[py:redis]`

## 5. Semantic search within a collection

Status: open
**Issue:** [#23](https://github.com/aukern/scout-api/issues/23)

### What to build
A **Search**: free text in, ranked **Results** out, scoped to one Collection. Embed the query,
pgvector cosine NN over the Collection's Chunks, return Results by relevance (score + Source).
Only `ready` Sources searchable. Repeated identical searches cached. Guarantees: no
cross-Collection leakage (ADR 0001); non-`ready` Sources contribute nothing.

### Acceptance criteria
- [ ] Query returns Chunks ranked by semantic relevance, each with score + Source
- [ ] Results restricted to the queried Collection — no leakage
- [ ] Only Chunks from `ready` Sources appear
- [ ] Repeated identical searches cached; cache invalidates on content change
- [ ] Search only retrieves — never synthesizes

### Blocked by
- slice 3 (Process a source into searchable chunks)

> Note: no `[py:arq]` despite api+LLM — synchronous-read exception (single fast query embedding).

## 6. Ask a question, stream a cited answer

Status: open
**Issue:** [#24](https://github.com/aukern/scout-api/issues/24)

### What to build
A **Question** scoped to one Collection: retrieve relevant Chunks → synthesize an **Answer** with
an LLM → **stream** it token-by-token with **Citations** to the Sources used. Insufficient context
is stated, not fabricated. Prototype prompt: "Answer using ONLY the numbered context. Cite sources
inline as [1], [2]. If insufficient, say so."

### Open question (FINDINGS)
- Citation granularity — point to the Source, or the specific Chunk?

### Acceptance criteria
- [ ] Scoped Question retrieves Chunks and synthesizes an Answer
- [ ] Answer streams incrementally
- [ ] Answer carries Citations to the Sources used
- [ ] Grounded only in retrieved context; insufficient context stated, not fabricated
- [ ] Retrieval is Collection-scoped

## 7. Research sessions

Status: open
**Issue:** [#19](https://github.com/aukern/scout-api/issues/19)

### What to build
A **Session** is a research workspace opened against exactly one Collection that records an
activity trail: every Search and Question (with its Answer). Recording is **opt-in** — only when a
Search/Question names a Session; otherwise transient. Open / list / get / close. Slices 5 and 6
record into a Session when given a session id; this slice owns the Session lifecycle + recording.

### Open question (FINDINGS)
- Session retention / cleanup policy

### Acceptance criteria
- [ ] Open a Session against exactly one Collection
- [ ] A Search/Question naming a Session is recorded; one that doesn't is not
- [ ] All activity scoped to that Session's Collection
- [ ] Sessions listed, fetched, closed
